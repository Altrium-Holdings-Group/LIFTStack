# LIFTStack Operating Model

## Role

LIFTStack is the engineering, implementation, automation, and tooling arm of the ecosystem.

## Platform posture

- WordPress: landing, SEO, trust, and public discovery CTA.
- GitHub: operating system, schemas, receipts, routing rules, implementation workflows, and deployment history.
- Genesis Command Center: control plane and build posture visibility.
- Genesis AI: classification, prioritization, summarization, and action generation only when it creates operational value.

## First principles

1. Do not cross-route discovery CTAs between brands.
2. Keep LIFTStack prospects in the technical implementation lane unless a governed handoff is created.
3. Create receipts for intake, routing, engineering decisions, implementation milestones, and deployment proof.
4. Prioritize reliability, auditability, logs, and measurable outcomes over novelty.
5. Treat technical rescue and platform migration as high-risk work requiring clear operator ownership.

## Current foundation

This repo currently includes:

- LIFTStack brand config
- no-cross-route discovery guard
- technical discovery intake schema
- discovery receipt writer
- Command Center sync client stub
- health endpoint
- discovery endpoint
- receipt contract endpoint
- Genesis context endpoint
- deployment and quality-gate scaffolding
